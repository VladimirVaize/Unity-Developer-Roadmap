# 🔍 Профилирование в Unity: Поиск узких мест (CPU, GPU, Memory, Render)

Этот материал посвящён встроенному инструменту Profiler (Профилировщик) в Unity. 
Он позволяет анализировать производительность игры в реальном времени, находить «бутылочные горлышки» (bottlenecks) и оптимизировать CPU, GPU, память и рендеринг.

---

## 🧠 Что такое профилирование?
Профилирование — это процесс измерения ресурсов, которые потребляет ваша игра: времени процессора (CPU), 
видеокарты (GPU), объёма оперативной памяти (RAM), видеопамяти (VRAM) и количества вызовов отрисовки (draw calls). 
Без профилирования оптимизация — это «гадание на кофейной гуще».

---

## 📊 Основные вкладки Profiler
Profiler открывается через `Window → Analysis → Profiler` (или `Ctrl+7`).

### 1. CPU Usage (загрузка процессора) 🖥️
#### Что показывает:
Время выполнения каждого метода в вашей игре: `Update()`, `FixedUpdate()`, `LateUpdate()`, физика, анимация, UI, сборка мусора (GC) и т.д.

#### Как использовать:
- Запустите игру (Play) и посмотрите на шкалу времени (timeline).
- Красные пики — «просадки» кадров (спайки).
- Разверните вкладку `CPU Usage` → выберите любой кадр → ниже появится иерархический список вызовов методов с временем выполнения (ms).
- Ищите методы, которые занимают больше всего времени (сортировка по `Total` или `Self`).

#### Пример узкого места:
Метод `Update()` в вашем скрипте вызывает `FindObjectOfType()` каждый кадр — это медленно. 
В профайлере вы увидите, что `FindObjectOfType` занимает 8 мс из 16 мс кадра (цель — 60 FPS = ~16.6 мс на кадр). 
Решение: кэшировать результат в `Start()`.

### 2. GPU Usage (загрузка видеокарты) 🎮
#### Что показывает:
Время, которое GPU тратит на рендеринг кадра: вершинные шейдеры, фрагментные шейдеры, заполнение экрана (fill rate), постэффекты.

#### Как использовать:
- Включите вкладку `GPU Usage` (в старых версиях Unity может быть `Rendering`).
- Запустите игру. Если время GPU больше, чем время CPU (например, GPU = 25 мс, CPU = 12 мс) — узкое место в видеокарте.
- Обратите внимание на `Draw Calls`, `Dynamic Batches`, `Shadow Casters`.

#### Пример узкого места:
У вас 500 объектов с разными материалами (500 draw calls) и сложными шейдерами. GPU тратит 30 мс на кадр. 
Решение: объединить материалы (текстурный атлас), использовать GPU Instancing или уменьшить количество источников теней.

### 3. Memory (память) 💾
#### Что показывает:
Распределение памяти: текстуры, меши, анимации, аудиоклипы, скриптовые объекты (Managed Memory). 
Также отображает сборку мусора (GC), которая может вызывать зависания.

#### Как использовать:
- Вкладка `Memory` → режим `Simple` или `Detailed`.
- Смотрите `Total Used Memory` и `GC Allocation`.
- Категории: `Assets` (текстуры, меши) и `MonoHeap` (память управляемого кода C#).
- Кнопка `Take Sample` — сделать снимок памяти в конкретный момент.

#### Пример узкого места:
Каждый раз, когда враг умирает, вы создаёте новый объект `List<int>` во взрыве. 
В профайлере вы видите частые пики `GC.Alloc` (выделение памяти), а затем спайк `GarbageCollect` (зависание на 50 мс). 
Решение: использовать пул объектов (Object Pool) и избегать создания мусора в `Update()`.

### 4. Rendering (рендеринг) 🖼️
#### Что показывает:
Детальную статистику по отрисовке: `SetPass Calls`, `Draw Calls`, `Batches`, `Triangles`, `Vertices`, использование теней и окклюзии.

#### Как использовать:
- Вкладка `Rendering` (иногда объединена с `GPU Usage`).
- Запустите игру и смотрите на количество `Draw Calls` и `Batches`.
- Чем меньше — тем лучше. Для мобильных устройств цель: <100 Draw Calls.
- Обратите внимание на `Saved By Batching` (статические/динамические батчи).

#### Пример узкого места:
В игре 2000 объектов с одинаковым материалом, но статический батчинг не включён. Draw Calls = 2000. 
После включения `Static Batching` (галочка на объектах) или `GPU Instancing` Draw Calls падают до 20. FPS растёт с 30 до 60.

## 🚦 Как искать узкие места (практический алгоритм)
1. Запустите Profiler (Play) и записывайте данные<br>
   Кнопка `Record` должна быть включена.

2. Найдите самый медленный кадр (красный пик)<br>
   Кликните на него — справа появится детализация.

3. Сравните CPU vs GPU<br>
   Если CPU выше — оптимизируйте скрипты, физику, GC. Если GPU выше — уменьшайте сложность графики (шейдеры, текстуры, тени, геометрию).

4. Смотрите на глубокий вызов (deep profiling)<br>
   Включите `Deep Profile` (замедляет игру, но даёт полную картину). Найдите метод с наибольшим `Total ms`.

5. Проверьте память<br>
   Нет ли утечек (память постоянно растёт)? Нет ли частого GC?

6. Используйте Frame Debugger (Window → Analysis → Frame Debugger) вместе с Profiler, чтобы увидеть каждый draw call.

## 📌 Типичные узкие места и их решения

| Компонент | Симптом в Profiler | Решение |
|---------|--------------------|--------------------------------------|
| CPU | `Scripts` > 5 мс | Оптимизировать код: убрать `FindObjectOfType`, `GetComponent` в Update |
| CPU | `GarbageCollect` спайки | Пул объектов, массивы фиксированного размера |
| GPU | `Rendering` ≈ 20+ мс | Меньше теней, LOD, текстуры меньшего размера |
| Memory | `MonoHeap` растёт бесконечно | Утечки: отписка от событий, очистка коллекций |
| Rendering | `Draw Calls` > 500 (мобилы) | Batching, Atlas, GPU Instancing |
