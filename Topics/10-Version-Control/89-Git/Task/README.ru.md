# 🎯 Задача: «Настройка Git-репозитория для командной разработки»
Вы — технический лидер команды из 5 разработчиков, работающих над игрой "Space Odyssey". 
Вам нужно настроить Git-репозиторий для комфортной командной работы, избегая типичных проблем с конфликтами сцен, префабов и большими файлами.

## 📝 Что нужно сделать:
### Часть 1: Настройка .gitignore и .gitattributes
1. Создайте правильный `.gitignore` для Unity-проекта:
   - Исключите: `Library/`, `Temp/`, `Logs/`, `UserSettings/`
   - Исключите: `*.csproj`, `*.sln`, `.vs/`
   - Сохраните: `Assets/`, `Packages/`, `ProjectSettings/`
  
2. Создайте `.gitattributes` для настройки:
   - Smart Merge для `*.unity`, `*.prefab`, `*.asset`
   - Git LFS для изображений, моделей, звуков, видео
  
### Часть 2: Настройка Git LFS
3. Установите Git LFS и настройте отслеживание:
   - Все PNG/JPG/TGA в папке `Assets/Textures/`
   - Все FBX/OBJ в папке `Assets/Models/`
   - Все WAV/MP3 в папке `Assets/Audio/`
   - Все MP4/MOV в папке `Assets/Videos/`
  
4. Напишите скрипт `migrate_to_lfs.sh` (bash) или `migrate_to_lfs.bat` (cmd) для автоматической миграции существующих файлов в LFS.

### Часть 3: Настройка UnityYAMLMerge
5. Настройте UnityYAMLMerge для всех разработчиков:
   - Напишите инструкцию в `README.md`
   - Добавьте путь к UnityYAMLMerge в зависимости от версии Unity
   - Проверьте, что Smart Merge работает на тестовом конфликте
  
### Часть 4: Работа с ветками
6. Создайте правила работы с ветками:
   - `main` — стабильная версия (только через Pull Request)
   - `develop` — активная разработка
   - `feature/*` — новые функции
   - `hotfix/*` — срочные исправления
  
7. Напишите в `CONTRIBUTING.md` инструкцию:
   - Как создавать новые ветки
   - Как делать коммиты (`feat:`, `fix:`, `docs:`)
   - Как разрешать конфликты в сценах и префабах
  
### Часть 5: Настройка хуков (Hooks)
8. Создайте pre-commit hook, который:
   - Проверяет, что файлы больше 50 МБ добавлены через LFS
   - Проверяет, что коммит содержит описание
   - Проверяет, что в коммите нет `Library/` или `Temp/`
  
9. Создайте post-merge hook, который:
    - Запускает `git lfs pull` после слияния
    - Выводит список изменённых сцен/префабов
  
### Часть 6: Тестирование конфликтов
10. Создайте симуляцию конфликта:
    - Два разработчика изменяют один префаб `Player.prefab`
    - Один изменяет позицию, второй — масштаб
    - Попробуйте смержить изменения с Smart Merge
    - Разрешите конфликт вручную (если Smart Merge не справился)
   
11. Создайте симуляцию конфликта сцены:
    - Два разработчика изменяют `Level1.unity`
    - Один добавляет новый GameObject, второй изменяет освещение
    - Попробуйте смержить через Smart Merge
   
### Часть 7: Документация
12. Напишите `README.md` для репозитория, содержащий:
    - Инструкцию по клонированию (`git clone + git lfs pull`)
    - Правила работы с Git
    - Как разрешать конфликты
    - Контакты для вопросов
   
---

## 🧰 Требования к реализации:
- Все настройки должны быть описаны в документации
- Хуки должны быть написаны на bash (для Mac/Linux) и batch (для Windows)
- Проверьте, что `.gitignore` и `.gitattributes` корректно работают
- Конфликты должны разрешаться без потери данных
- Размер репозитория должен оставаться < 200 МБ (без LFS)

## 🔍 Проверка работоспособности:

2. Проверьте, что `Library/` отсутствует в репозитории
3. Проверьте, что LFS-файлы скачиваются через `git lfs pull`
4. Проверьте, что Smart Merge работает при мерже сцен
